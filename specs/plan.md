## Phase 3: Service Layer

### 3.1 `app/config.py`

Pydantic `BaseSettings` class loading from `.env`. Exposes typed fields for all env vars. Singleton pattern via `@lru_cache`.

### 3.2 `app/database.py`

Creates the SQLAlchemy engine with `check_same_thread=False` for SQLite. Provides `get_db()` generator for FastAPI `Depends` injection and `create_all()` for initial schema setup.

### 3.3 `app/services/strava_service.py` — Strava Sync Service

- **`build_authorization_url()`** — generates the Strava consent URL with scope `activity:read_all`.
- **`exchange_code(code)`** — POSTs to `https://www.strava.com/oauth/token` and stores access + refresh tokens.
- **`refresh_access_token()`** — auto-refreshes when `expires_at < now()`. Called before every API request.
- **`sync_week_activities(week_id)`** — fetches `/athlete/activities?after=&before=` filtered to the week window, upserts `StravaActivity` rows using `strava_id` as the dedup key.
- **`compute_weekly_aggregates(week_id)`** — queries `StravaActivity` and returns total moving time, total distance, and session counts per activity type.

### 3.4 `app/services/prompt_engine.py` — Contract-First Prompt Builder

Gathers all relevant data for a `week_id` and assembles a deterministic JSON payload (the "Contract") before touching the LLM:

```json
{
  "week": { "start": "2025-01-06", "end": "2025-01-12" },
  "targets": { "calories": 2200, "protein_g": 180 },
  "measurements": { "weight_kg": 84.5, "weight_prev_kg": 85.1 },
  "daily_logs": [ { "date": "...", "adherence": 8, "calories_in": 2150 } ],
  "strava_aggregates": { "run_km": 22.3, "strength_sessions": 3 }
}
```

The payload is JSON-serialized and SHA-256 hashed for idempotency checks. This hash is stored in `weekly_evals.input_hash`.

### 3.5 `app/services/llm_client.py` — LM Studio Client

- Uses `httpx.AsyncClient` targeting `LM_STUDIO_BASE_URL`.
- Calls `POST /v1/chat/completions` with `response_format: {"type": "json_object"}` to enforce structured output.
- Passes the system prompt from `app/prompts/system_prompt.txt` and the contract JSON as the user message.
- Returns the raw JSON string; validation is handled by the caller (Pydantic parse).
- Implements exponential backoff retry (max 3 attempts) for connection errors.

## Phase 4: Evaluation Logic & Guardrails

### 4.1 `app/schemas/eval_output.py` — Strict LLM Output Schema

Pydantic model defining the exact JSON structure the LLM must return:

```python
class EvalOutput(BaseModel):
    overall_score: int                        # 1-10
    summary: str                              # 2-3 sentence narrative
    wins: list[str]                           # Concrete achievements
    misses: list[str]                         # Gaps vs targets
    nutrition_analysis: NutritionAnalysis
    training_analysis: TrainingAnalysis
    recommendations: list[Recommendation]    # max 5
    data_confidence: float                    # 0.0-1.0, based on log completeness
```

### 4.2 `app/services/evidence_collector.py`

After the LLM responds, this service cross-references each claim in `EvalOutput` against the database:

- For nutrition claims → links to `DailyLog` rows for the week.
- For training claims → links to `StravaActivity` rows (by `strava_id`).
- For weight/measurement claims → links to `WeeklyMeasurement` row.

Produces an `evidence_map` dict stored in `weekly_evals.evidence_map_json` for auditability.

### 4.3 `app/prompts/system_prompt.txt`

Carefully engineered system prompt enforcing:

- **Role:** "You are a supportive fitness coach analyzing a user's weekly data."
- **Format constraint:** "You MUST respond only with a JSON object matching the provided schema. No prose outside JSON."
- **Safety guardrail:** "You are NOT a medical professional. Do NOT provide medical diagnoses, prescribe medications, or advise on injuries beyond recommending rest."
- **Tone directive:** "Be collaborative and encouraging. Frame misses as opportunities, not failures."
- **Evidence directive:** "All claims must be grounded in the data provided. Do not infer trends beyond one week unless prev_week data is present."

## Phase 5: API Endpoints & Repeatability

### 5.1 `app/main.py`

FastAPI app factory. Registers all routers, configures CORS, runs `create_all()` on startup.

### 5.2 `app/api/auth.py` — Strava OAuth

| Endpoint | Description |
|----------|-------------|
| `GET /auth/strava` | Redirects user to Strava consent page |
| `GET /auth/strava/callback` | Handles code exchange, stores tokens, redirects to app |

### 5.3 `app/api/logs.py` — Manual Data Entry

| Endpoint | Description |
|----------|-------------|
| `POST /logs/daily` | Create or update a `DailyLog` entry for a given date |
| `GET /logs/daily/{date}` | Retrieve a specific day's log |
| `POST /logs/weekly` | Create or update a `WeeklyMeasurement` record |
| `GET /logs/weekly/{week_start}` | Retrieve a week's measurements + daily logs |
| `POST /logs/targets` | Set new `PlanTargets` (creates versioned record) |

### 5.4 `app/api/strava.py` — Strava Sync

| Endpoint | Description |
|----------|-------------|
| `POST /strava/sync/{week_start}` | Trigger Strava sync for the given week, upserts activities |
| `GET /strava/activities/{week_start}` | List synced Strava activities for the week |

### 5.5 `app/api/evaluate.py` — Core Evaluation Endpoint

**`POST /evaluate/{week_id}`** — Full idempotency pipeline:

1. Call `prompt_engine.build_contract(week_id)` to assemble the data payload.
2. Compute SHA-256 hash of the serialized contract JSON.
3. Query `weekly_evals WHERE week_id = ? AND input_hash = ?`
4. If a matching row exists → return cached `EvalOutput` immediately (no LLM call).
5. Otherwise → call `llm_client.generate(contract)`, parse via `EvalOutput` Pydantic model.
6. Run `evidence_collector.map_claims(eval_output, week_id)`.
7. Persist new `WeeklyEval` row with hash, raw response, parsed JSON, evidence map.
8. Return the `EvalOutput` to the caller.

**`GET /evaluate/{week_id}`** — Retrieve the most recent evaluation without re-running.

## Key Design Decisions

### Why Contract-First Prompt Engineering?

Serializing all data to a structured JSON before calling the LLM guarantees: (1) reproducibility via deterministic hashing, (2) a clear boundary between data retrieval and AI reasoning, and (3) easier debugging since the exact model input is always inspectable.

### Why SHA-256 Input Hashing for Idempotency?

LLM calls are expensive and non-deterministic. By hashing the input contract, we can return the cached evaluation if the underlying data hasn't changed — avoiding redundant calls and ensuring consistent reports for the same week.

### Why `raw_json` on `StravaActivity`?

Strava's API schema evolves. Storing the raw payload allows reprocessing historical activities as new fields become relevant, without requiring a re-sync from Strava.

### Why a Separate `evidence_collector.py`?

The evidence mapper runs post-generation and independently of the LLM. This means evaluation claims can be audited against the source data even if the LLM hallucinated a specific number, enabling a human review layer.
